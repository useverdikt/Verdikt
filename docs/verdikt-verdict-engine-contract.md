# Verdikt Verdict Engine — Conformance Contract

This document is the authoritative statement of the behavioural properties Verdikt guarantees. It is the source of truth for auditors, prospects, and internal engineering decisions. When the implementation changes, this document must be updated to reflect the enforcement mechanism explicitly.

---

## Property 1 — Signal ingestion is idempotent per idempotency key

Submitting the same signal payload twice with the same `X-Idempotency-Key` header (or `body.idempotency_key`) produces exactly one write to the `signals` table and exactly one verdict evaluation.

**Enforcement:** The existence check and the `INSERT` are co-located inside the same database transaction. A `SELECT ... LIMIT 1 WHERE idempotency_key = $key` runs first under transaction isolation; if any matching row exists the function returns early without inserting. The `INSERT` additionally carries `ON CONFLICT DO NOTHING` backed by a partial unique index `idx_signals_idempotency ON signals(release_id, signal_id, idempotency_key) WHERE idempotency_key IS NOT NULL`. Both the transaction-scope check and the DB-level constraint must be violated for a duplicate to land — the combination is race-proof under any supported isolation level.

---

## Property 2 — Verdict is computed from frozen inputs

The verdict at certification time is computed from the signal values and threshold map that were current at the moment of certification, not from any subsequently mutated state.

**Enforcement:** `persistCertificationSnapshot` freezes both the threshold map and the signal map into `certification_snapshots` at certification time. `buildGateContext` reads from this snapshot, not from the live `thresholds` table, for any release that has already been certified. Threshold updates after certification do not retroactively change a certified release's verdict.

---

## Property 3 — Threshold and signal-definition changes are human-gated

No agent API key may create, update, or delete thresholds, signal definitions, integration credentials, or workspace policies.

**Enforcement:** All control-plane mutation routes (`POST /thresholds`, `POST /signal-definitions`, `PUT/DELETE /signal-integrations`, `POST /signal-csv-imports`, `POST /policies`, `PUT /baseline-policy`, `PUT/DELETE /outbound-webhook`) carry the `requireHumanSession` middleware, which returns HTTP 403 for any request authenticated via `authType: "api_key"`.

## Property 4 — Overrides require a human with approver authority

Certifying an uncertified release via the override path requires a human session with a role in `{vp_engineering, cto, org_admin}`. No API key may approve an override.

**Enforcement:** The override route carries both `requireHumanSession` (blocks any `api_key` auth type, regardless of role) and `requireOverrideApproverRole` (checks `req.auth.role` against `OVERRIDE_APPROVER_ROLES`). Both checks must pass. `requireHumanSession` is the authoritative guard; `requireOverrideApproverRole` is the role filter applied after session type is confirmed.

## Property 5 — SQL queries use correctly-typed parameter bindings

All `IN (...)` clauses and multi-parameter queries use sequentially-numbered placeholders (`$1`, `$2`, `$3`, ...) generated by `array.map((_, i) => \`$${i + N}\`)`. Repeating a single placeholder (e.g. `map(() => "$2")`) is a Postgres protocol violation that causes a runtime crash.

**Enforcement:** Verified by code review and regression test (see `backend/test/gateEnforcement.test.js` and `backend/test/p0Fixes.test.js`). No instances of the `.map(() => "$N")` anti-pattern exist in the codebase as of this contract version. If a helper `buildInClause(count, startIdx)` is introduced, this becomes the enforced pattern.

## Property 6 — The audit chain is append-only and fork-free

Every audit event written to `audit_events` is linked to the previous event for the same workspace via a SHA-256 hash chain (`prev_hash → row_hash`). No two rows for the same workspace may share the same `prev_hash` value. The chain must be recomputable from genesis and must verify clean via `verifyAuditIntegrity`.

**Enforcement:** Audit chain writes are serialised per-workspace via a PostgreSQL transaction-scoped advisory lock: `SELECT pg_advisory_xact_lock(hashtext(workspace_id))`. This lock is acquired as the **first statement** in the transaction, before the chain tip is read, so no concurrent writer can interleave between the tip read and the INSERT. The advisory lock, the chain tip `SELECT`, the hash computation, and the `INSERT INTO audit_events` are all within a single transaction. The lock is released automatically on `COMMIT` or `ROLLBACK`.

The single code path that writes to `audit_events` is `writeAudit` in `backend/src/services/audit.js`. All other services call `writeAudit`; none write directly to `audit_events`. This is verified by grep — the string `INSERT INTO audit_events` appears exactly once in the codebase.

**Pre-existing forks (test database):** The local test database (`verdikt_test`) contains forked records written before this enforcement mechanism was in place, primarily from concurrent load testing. To detect forks in any environment, run:

```sql
SELECT workspace_id, prev_hash, COUNT(*) AS cnt
FROM audit_events
WHERE prev_hash IS NOT NULL
GROUP BY workspace_id, prev_hash
HAVING COUNT(*) > 1
ORDER BY cnt DESC;
```

If rows are returned for production workspaces, those forks predate this fix. The recommended remediation is to log them as `AUDIT_CHAIN_FORK_DETECTED` events in the audit log itself (append-only acknowledgement), not to mutate existing rows (which are append-only by policy). The fix ensures no new forks are created.

## Property 7 — Agent keys are scoped to data-plane operations only

Agent API keys (`vdk_live_...`) may post signals, pull integration data, query release state, and check the gate. They may not mutate credentials, membership, thresholds, policies, definitions, or any other control-plane resource.

**Enforcement:** See Property 3 (human-session-gated routes). Agent keys always receive `role: "agent"` in the auth context, regardless of the issuing user's role. This means they also fail `requireOverrideApproverRole` independently of `requireHumanSession`.
